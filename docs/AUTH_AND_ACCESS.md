# Aperture — Авторизация, роли и контроль доступа

План по аутентификации, авторизации, ролям и связанным механизмам.

---

## 1. Аутентификация (Authentication)

**Вопрос:** как клиент доказывает, что он тот, за кого себя выдаёт?

### Варианты

| Способ | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| **API Key** (Bearer) | Просто, привычно для LLM API | Хранение ключей, ротация | MVP, большинство сценариев |
| **API Key** (Header `X-API-Key`) | Альтернатива Bearer | То же | Если нужна совместимость с разными клиентами |
| **JWT** | Stateless, claims, expiry | Сложнее, нужен issuer | Интеграция с существующей IdP |
| **Basic Auth** | Очень просто | Слабо для prod | Только для dev/test |
| **OAuth2 / OIDC** | Enterprise, SSO | Сложная настройка | Корпоративные клиенты |

### Рекомендация по фазам

| Фаза | Реализация |
|------|------------|
| **MVP** | Один API key из env — все запросы проходят (или отключено) |
| **v0.2** | Множественные API keys из конфига/файла, привязка к ключу |
| **v0.3** | JWT как опция (для интеграции с Keycloak, Auth0 и т.п.) |
| **Позже** | OAuth2/OIDC для enterprise |

### Формат запроса (OpenAI-совместимо)

```
Authorization: Bearer sk-aperture-xxx
# или
X-API-Key: sk-aperture-xxx
```

## 2. Авторизация и роли (Authorization)

**Вопрос:** что разрешено делать после аутентификации?

### Роли (предложение)

| Роль | Описание | Права |
|------|----------|-------|
| **admin** | Администрирование | Всё + управление ключами, конфигом, просмотр метрик |
| **user** | Обычный пользователь | Вызов `/v1/chat/completions`, `/v1/models` в рамках своих лимитов |
| **readonly** | Только чтение | `/v1/models`, возможно просмотр usage (без вызова LLM) |
| **service** | Сервисный аккаунт | Как user, но для машин/интеграций, отдельные лимиты |

### Разрешения (permissions)

Помимо ролей — точечные ограничения:

| Разрешение | Описание |
|------------|----------|
| `models:list` | Доступ к `/v1/models` |
| `chat:complete` | Вызов chat completions |
| `embeddings:create` | Вызов embeddings (если добавим) |
| `models:allowed` | Список разрешённых моделей (иначе — все) |
| `providers:allowed` | Список разрешённых провайдеров |

### Привязка к ключу

```
key_id: sk-aperture-abc123
  role: user
  permissions: [models:list, chat:complete]
  allowed_models: [gpt-4o-mini, gpt-4o]   # пусто = все
  rate_limit: 60/min
  budget_daily: 10.00   # USD
```

## 3. Ключи (API Keys)

### Жизненный цикл

| Действие | MVP | v0.2 | Позже |
|----------|-----|------|-------|
| Создание | Ручное в конфиге | Конфиг/файл | Admin API |
| Хранение | Plain text в env | Файл, env | Hash в БД (опционально) |
| Ротация | Ручная | Ручная | Admin API + TTL |
| Отзыв | Удалить из конфига | Удалить | Revoke через API |

### Формат ключа (для распознавания)

```
sk-aperture-<random>   — ключ Aperture (наш)
sk-proj-xxx            — можно пропускать «как есть» в upstream (passthrough)
```

## 4. Мультитенантность

**Вопрос:** изоляция данных между клиентами (организациями, проектами).

| Уровень | Описание | Сложность |
|---------|----------|-----------|
| **Нет** | Все ключи в одном пространстве | Просто |
| **Soft** | Ключи с тегом `tenant_id`, отчётность по тенанту | Средне |
| **Hard** | Полная изоляция, отдельные лимиты/бюджеты на тенанта | Сложнее |

**Рекомендация:** на старте — soft (tenant_id как атрибут ключа). Hard — при явном спросе.

## 5. Middleware pipeline (порядок обработки)

```
Request
   │
   ▼
┌──────────────┐
│ 1. Logging   │  — записать входящий запрос
└──────┬───────┘
       ▼
┌──────────────┐
│ 2. Auth      │  — извлечь ключ, проверить, положить в context
└──────┬───────┘
       ▼
┌──────────────┐
│ 3. Authorize │  — проверить роль/разрешения для эндпоинта
└──────┬───────┘
       ▼
┌──────────────┐
│ 4. Rate limit│  — проверить лимиты по ключу
└──────┬───────┘
       ▼
┌──────────────┐
│ 5. Budget    │  — проверить дневной/месячный лимит (если есть)
└──────┬───────┘
       ▼
┌──────────────┐
│ 6. Router    │  — выбрать провайдер, отправить запрос
└──────┬───────┘
       ▼
┌──────────────┐
│ 7. Cost track│  — учесть токены/стоимость
└──────┬───────┘
       ▼
Response
```

## 6. Audit / Логирование

Что логировать для безопасности и отладки:

| Событие | Данные |
|---------|--------|
| Успешный запрос | key_id (masked), model, tokens, timestamp |
| Отказ auth | причина (invalid key, expired) |
| Отказ rate limit | key_id, limit |
| Отказ budget | key_id, usage, limit |
| Admin действия | кто, что, когда (если есть admin API) |

**Важно:** не логировать тело запросов (сообщения чата) — персональные данные.

## 7. Обновлённый план разработки

### Фаза 1 (MVP) — без auth
- Auth отключена или один статический ключ из env
- Цель: проверить proxy

### Фаза 2 — базовая auth
- Множественные API keys из конфига
- Проверка ключа в middleware
- Привязка запроса к key_id для rate limit и cost tracking

### Фаза 3 — роли и разрешения
- Роли: admin, user, readonly
- Проверка permissions на эндпоинт
- allowed_models, allowed_providers

### Фаза 4 — управление ключами
- Admin API: создать/отозвать ключ (опционально)
- Или оставить конфиг-файл для простоты

### Фаза 5 (опционально)
- JWT
- Мультитенантность (tenant_id)
- OAuth2/OIDC

## 8. Конфиг (пример)

```yaml
auth:
  enabled: true
  keys:
    - id: sk-aperture-dev123
      name: "Development"
      role: admin
      # permissions: [models:list, chat:complete, admin]
    - id: sk-aperture-user456
      name: "App Backend"
      role: user
      allowed_models: [gpt-4o-mini]
      rate_limit: 100/min
      budget_daily: 5.00
```
